## C2. 产品形态与定位

传统的 `text-transfer` 是一次性、点对点的加密投递；而 `online-clipboard` 的定位更接近一个**持续存在的聊天窗口**或**共享记事本**。

| 维度             | Text Transfer                        | Online Clipboard                       |
| :--------------- | :----------------------------------- | :------------------------------------- |
| **连接方式**     | 扫码配对，一对一                     | URL 直接加入，多对多                   |
| **设备数量**     | 限 2 台                              | 同房间最多 10 台                       |
| **内容模式**     | 单条收发                             | 多条追加，对话式叠加                   |
| **持久性**       | 会话结束即销毁                       | 房间有效期内持续存在（24h），可断线恢复 |
| **通信方式**     | WebRTC P2P 优先                      | WebSocket 为主（多端广播需中心节点）   |
| **加密方式**     | 强制 E2EE                            | 默认不加密，可选 E2EE 模式             |
| **使用场景**     | 极速传一段私密文字/单张图            | 持续性跨设备剪贴板 / 团队协作对话流   |

**产品心智隔离：**
- **工具 A (Quick Send)**：主打极简、无痕，强调**“阅后即焚、极速投递”**。
- **工具 C (Shared Notepad)**：主打上下文、留存，强调**“多端协同、持续纪录”**。建议在 A 的主截面提供一个跨平台升级入口“需要来回对话？[升级为共享房间]”，防范用户的概念模糊。

**核心特点：**

- **房间制 (Clipboard Room)**：每个共享面板由一个特定的 URL 定义（例：`/tools/clipboard/r/abc123`）。
- **多端流式同步**：任何设备打开同一房间链接，都能实时看到面板上的内容。不仅是单次发送，还能实现**多次追加输入**。
- **对话式追加体验**：每次输入的独立内容像聊天记录一样一条条叠加，两端同时可见、同时更新。
- **半持久化记录**：房间未过期前（24 小时），同一设备或新设备输入相同房间链接，仍能看到之前的剪贴流历史。
- **一键清空**：自带一键清空整个房间画板功能，实现用完即删。

## C3. 房间机制

### C3.1 房间 URL 结构

```
https://toolport.dev/tools/clipboard/r/<RoomID>
https://toolport.dev/tools/clipboard/r/<RoomID>#<AES-Key-Base64URL>   ← E2EE 模式
│                                       │       │
│                     RoomID (6位字母数字) │    (可选) AES-256 密钥
│
└── 工具路由（与工具 A 的 /r/ 子路径风格一致）
```

* **RoomID：** 6 位字母数字（`[a-z0-9]`），由创建者前端随机生成，~21.7 亿组合。
* **语言无关：** 房间 URL 不含语言前缀，UI 语言由各端自身偏好决定。
* **房间号可读输入：** 用户也可以直接在入口页手动输入 6 位房间号加入（非加密房间下有效）。

### C3.2 房间生命周期

| 阶段           | 说明                                                                     |
| :------------- | :----------------------------------------------------------------------- |
| **创建**       | 用户点击 "新建剪贴板" → 前端生成 RoomID → 连接 WebSocket 注册房间。      |
| **活跃**       | 有至少 1 个连接在线，持续接收/广播消息。                                 |
| **空闲**       | 所有连接断开，进入倒计时（默认 30 分钟），期间重新加入可恢复历史。       |
| **过期销毁**   | 空闲超时 → 服务端销毁房间，清除所有暂存消息。                           |
| **最长生存**   | 无论是否活跃，免费房间最长存活 **24 小时**，到期强制销毁。               |

**【PLG / 意愿收集与功能票选】**
在 MVP 阶段，**绝对不要**在页面内放置“能看不能点”的置灰 `[✨Pro]` 按钮，这会激起初来乍到用户的反感。
替代体验方案：在侧边栏或汉堡菜单底端，提供一个醒目的 `💡 投票下一个功能 (Vote Next)`，弹窗内列出以下高级特性供用户票选：
* **永久存档室**：打破 24 小时销毁限制，化身为长期记事本。
* **定制短链接**：彻底摆脱枯燥的随机码（例如 `/r/mycompany`）。
* **最高级密码锁**：避免 URL 意外泄露导致的数据注水。
用户点击投票后顺水推舟：“感谢反馈！功能推出时我们将第一时间通知您 [输入邮箱]”，自然收集 Waitlist 并建立 Pro 转化的基础。

### C3.3 消息暂存

* 服务端（PartyKit）为每个房间维护**消息队列**，最多暂存最近 **100 条**消息。
* 存储方案：使用 **PartyKit 内置的房间状态**（基于 Durable Objects），天然适合有状态的房间模型。
* 新设备加入或页面刷新时，服务端推送全部暂存消息（非加密房间为明文，E2EE 房间为密文），客户端本地渲染。
* 超过 100 条时，最早的消息被淘汰（FIFO）。**防慌乱体验优化**：容量达到 80 条时，列表顶部出现浅色提示“容量将满，最早的记录即将覆盖”；当发生真实顶替淘汰时，顶部悬浮一条固定占位气泡：“💭 N 条历史记录已自然清理消失”，避免用户误以为是系统 Bug 导致的数据丢失。
* 一键清空：客户端发送 `CLEAR` 指令 → 服务端清除所有暂存消息 → 广播清空事件给所有连接。

## C4. 操作流程

### C4.1 创建房间

```
用户进入 /tools/clipboard 落地页
    │
    ├─ 选项一：点击 [新建剪贴板]
    │       │
    │       ├─ (可选) 勾选 "启用端到端加密" → 前端额外生成 AES-256 密钥
    │       │
    │       ├─ 前端生成 RoomID (6 位)
    │       │
    │       ├─ 连接 WebSocket (PartyKit)，发送 CREATE { roomId, peerId }
    │       │       │
    │       │       ├─ 成功 → 重定向到 /tools/clipboard/r/<RoomID>[#key]
    │       │       └─ RoomID 已存在 → 重新生成（自动重试 3 次）
    │       │
    │       └─ 页面展示：
    │               ├─ 🎯 首屏焦点：大号引导卡片"将链接发送到其他设备即可开始同步"
    │               │       ├─ [复制链接] 主按钮（最醒目）
    │               │       ├─ 房间二维码（手机扫码加入）
    │               │       └─ 引导文案在第一条消息发出后自动收起
    │               ├─ 聊天式消息列表（空）+ 底部输入框
    │               └─ [一键清空] 按钮
    │
    └─ 选项二：输入已有房间号 → 加入（见 C4.2）
```

### C4.2 加入房间

```
用户打开房间 URL（扫码 / 点击分享链接 / 手动输入房间号）
    │
    ├─ 从 URL 提取 RoomID（并强制 `.toLowerCase().trim()`，容错处理）；如有 Hash 则提取 AES 密钥
    │
    ├─ (E2EE 模式) history.replaceState 清除 Hash，密钥存入 sessionStorage
    │
    ├─ 连接 WebSocket，发送 JOIN { roomId, peerId }
    │       │
    │       ├─ 成功 → 服务端推送暂存消息 → 渲染历史
    │       ├─ 房间不存在 → 提示 "房间不存在或已过期"
    │       └─ 房间已满 (10人) → 提示 "房间已满"
    │
    └─ 进入房间页面，可立即发送和接收消息
```

### C4.3 实时持续交互

```
底部输入框打字，或直接粘贴截图/图片 → 点击发送 / Ctrl+Enter / Cmd+Enter
    │
    ├─ (若含截图/图片)
    │       ├─ 前端压缩至最大 200KB（canvas 缩放 + quality 调节，约 1280×720 清晰度）
    │       ├─ 压缩后在发送按钮旁显示缩略图预览 + 压缩比提示（如"800KB → 195KB"），用户确认后发送
    │       ├─ 转 Base64 编码后（~267KB）作为消息 content 发送
    │       ├─ 服务端按普通消息暂存（计入 100 条 FIFO 队列）
    │       └─ 接收端解码为图片，以聊天图文气泡展示 + [保存图片] 按钮
    │
    ├─ (E2EE 模式) 前端加密：AES-256-GCM(plaintext/imageBase64, key, randomIV)
    │
    ├─ 构造消息：{ peerId, seq, content/ciphertext, timestamp }
    │
    ├─ 发送到 WebSocket → 服务端广播给同房间所有其他连接 + 暂存
    │
    ├─ 本地立即渲染为新气泡（乐观更新，不等待服务端确认）
    │
    └─ 其他端实时收到 → 解密（如需） → 渲染为新气泡
```

**一致性保证**：不需要反复扫码，只要房间里有人，就可以持续交互输入文本。

## C5. 加密模式（可选 E2EE）

默认模式下，消息以明文经 WebSocket 传输（仍走 HTTPS/WSS 传输层加密），满足大部分日常跨设备复制粘贴需求。对于隐私敏感场景，可在创建房间时启用 E2EE。

### C5.1 默认模式（非 E2EE）

* 消息明文通过 WSS 传输，传输层由 HTTPS 保护。
* PartyKit 服务端可见消息内容（但不做任何存储分析，仅暂存用于历史回放）。
* 房间号即访问凭证，知道房间号即可加入查看。
* 用户可直接手动输入 6 位房间号加入。

### C5.2 E2EE 模式

创建房间时勾选 "启用端到端加密"，激活以下机制：

| 项目           | 说明                                                                         |
| :------------- | :--------------------------------------------------------------------------- |
| **算法**       | AES-256-GCM（与工具 A 一致，复用加密模块）。                                |
| **密钥生成**   | `crypto.getRandomValues()` 生成 256 位密钥。                                |
| **密钥分发**   | 通过 URL Hash（`#` 后），创建者将完整 URL 分享给其他人。                    |
| **密钥持久化** | 加载后 `history.replaceState` 清除 Hash；密钥存入 `sessionStorage`（标签页/窗口生命周期内有效，刷新不丢失，关闭标签页自动清除）。写入时 try-catch，iOS 隐私浏览模式下可能失败，降级到内存变量并提示"隐私模式下刷新页面将丢失加密密钥，建议切换到普通模式"。 |
| **每条消息**   | 独立随机 IV（12 字节）+ 密文 + AuthTag。                                    |
| **加密标识**   | 房间顶栏展示 🔒 加密徽章；非加密房间展示 🔓 标识。                          |

**安全边界：**
* E2EE 模式下，服务端零知识，仅转发和暂存密文。
* 知道 RoomID 但不知道密钥的人可连接 WebSocket 但只能看到密文。
* 手动输入房间号加入 E2EE 房间 → 能看到密文但无法解密 → 提示 "此房间已加密，请使用完整分享链接加入"。
* 密钥仅在分享 URL 时通过用户自选渠道传递（微信、短信等）。

**关于为何不默认启用 E2EE：**
> Online Clipboard 的核心场景是自己多设备间的快速文本复制（如手机复制到电脑），属于低隐私风险的日常操作。强制 E2EE 会带来：1) 手动输入房间号不可用（需完整 URL）；2) 密钥管理复杂度增加。对于敏感内容，Text Transfer（强制 E2EE + P2P）是更合适的工具。

## C6. 技术实现

### C6.1 服务端：PartyKit

| 项目             | 说明                                                                     |
| :--------------- | :----------------------------------------------------------------------- |
| **房间管理**     | PartyKit 原生房间模型，每个 RoomID 对应一个 Party 实例。                 |
| **状态存储**     | PartyKit 内置 storage API（基于 Durable Objects），存储消息队列和房间元数据。 |
| **消息广播**     | `room.broadcast()` 将消息推送给同房间所有连接。                          |
| **生命周期**     | Party `onClose` / `onAlarm` 管理空闲超时和最长生存时间。                 |

### C6.2 降级方案

| 场景               | 降级策略                                                               |
| :----------------- | :--------------------------------------------------------------------- |
| **PartyKit 额度耗尽** | 添加 `Max-Connections` 强校验，若全站总活动连接快触达免费上限（50），立即 HTTP 429 拒绝创建新房间，并提示用户"拥挤，请暂用单次投递(工具A)"。 |
| **WebSocket 不可用** | 自动切到 HTTP 长轮询（PartyKit 原生支持），界面提示"实时性降低"。     |
| **PartyKit 宕机**  | 降级到 KV 短轮询（2s 间隔），仅支持最新 10 条消息，界面提示降级状态。 |

## C7. WebSocket 通信协议

### C7.1 消息类型

```typescript
// 客户端 → 服务端
type ClientMessage =
  | { type: 'CREATE'; roomId: string; peerId: string; encrypted: boolean }
  | { type: 'JOIN'; roomId: string; peerId: string }
  | { type: 'REJOIN'; roomId: string; peerId: string; lastServerSeq: number }
  | { type: 'MESSAGE'; peerId: string; seq: number; content: string; timestamp: number }
      // content: 非加密时为明文，E2EE 时为 JSON { iv, ciphertext }
  | { type: 'CLEAR'; peerId: string }
  | { type: 'PING' }

// 服务端 → 客户端
type ServerMessage =
  | { type: 'CREATED'; roomId: string; expiresAt: number }
  | { type: 'JOINED'; roomId: string; peerCount: number; encrypted: boolean; history: Message[] }
  | { type: 'REJOINED'; roomId: string; peerCount: number; missedMessages: Message[] }
  | { type: 'PEER_JOINED'; peerId: string; peerCount: number }
  | { type: 'PEER_LEFT'; peerId: string; peerCount: number }
  | { type: 'MESSAGE'; peerId: string; seq: number; content: string; timestamp: number; serverSeq: number }
      // serverSeq: 服务端全局递增序号，用于统一排序
  | { type: 'CLEARED'; peerId: string }
  | { type: 'ERROR'; code: string; message: string }
  | { type: 'ROOM_EXPIRED' }
  | { type: 'PONG' }
```

### C7.2 消息排序

* 每个客户端维护本地 `seq` 递增序号（用于同一发送者的防重放校验）。
* 服务端为每条消息分配全局递增 `serverSeq`（用于跨发送者的统一排序）。
* 客户端按 `serverSeq` 排序展示消息，保证多端看到的顺序一致。

### C7.3 心跳与重连

* 心跳：**15 秒**一次 `PING`，连续 2 次无 `PONG`（30s）判定断线（收紧心跳以快速释放 PartyKit 连接额度）。
* **空闲踢：** 页面不可见（`visibilitychange` 事件，辅以 `pagehide` 兜底）超过 **15 分钟**后主动断开 WebSocket 以下降成本。页面重新可见时自动 REJOIN 增量同步。**注意：断线与重连过程在 UI 上必须完全静默处理**，即使重新连接期间也不要闪出红/黄色警告条，避免打断用户的沉浸对照体验。
* 重连：使用 `REJOIN` 消息，携带 `peerId` + 最后收到的 `serverSeq`，服务端仅推送缺失消息（增量同步）。
* 重连策略：指数退避（1s → 2s → 4s → 8s → 16s），最多 10 次。
* 超过最大重连次数 → 提示"连接已断开"+ 手动重连按钮。

## C8. 连接状态机

| 状态               | 说明                                                         |
| :----------------- | :----------------------------------------------------------- |
| **DISCONNECTED**   | 初始状态 / 断线状态。                                       |
| **CONNECTING**     | 正在连接 WebSocket。                                         |
| **SYNCING**        | 已连接，正在接收暂存历史消息。                               |
| **CONNECTED**      | 同步完成，正常收发中。                                       |
| **RECONNECTING**   | 断线后自动重连中（指数退避）。                               |
| **EXPIRED**        | 房间已过期销毁，不可恢复。                                   |

## C11. 安全与防滥用

| 机制               | 规则                                                                     |
| :----------------- | :----------------------------------------------------------------------- |
| **传输层加密**     | 全链路 WSS/HTTPS，即使非 E2EE 模式也不会明文暴露在网络上。              |
| **E2EE（可选）**   | AES-256-GCM，服务端零知识，仅转发密文。密钥存 sessionStorage。          |
| **房间人数限制**   | 每房间最多 10 个连接，超出拒绝（E203）。                                |
| **消息大小限制**   | 纯文本单条最大 64KB；图片前端压缩至 200KB 后 Base64 编码（~267KB），适配 Durable Objects 128KB 值上限。 |
| **消息频率限制**   | 单连接每秒最多 5 条消息。                                                |
| **房间生命周期**   | 空闲 30 分钟或最长 24 小时后强制销毁。                                  |
| **暂存上限**       | 每房间最多暂存 100 条消息，超出 FIFO 淘汰。                             |
| **IP 频率限制**    | 单 IP 每分钟最多创建 5 个房间，防滥用。                                  |
| **人机验证**       | 异常高频创建房间时触发 Cloudflare Turnstile。                           |
| **空内容防护**     | 输入为空时发送按钮 disabled。                                            |
| **断线恢复**       | 未发送内容暂存 localStorage，重连后可恢复。                              |
| **内容合规风控**   | 若未开启 E2EE (即公开分享)，界面需提供极简 🚨 Report 按钮。一旦同个 RoomID 累计收到 3 个异地 IP 违规举报，PartyKit 自动解散房间，彻底规避涉黄防黑产等带来的 DMCA/封禁连带责任。 |
| **访问控制限制**   | 知道房间 URL 即拥有完全权限，无踢人/撤销机制。V2.1 可考虑房间密码或创建者管理权限。 |

## C14. PWA 与留存转化策略 (Growth)

作为极为高频的“跨端工具”，本项目天然适合 Progressive Web App。
* **桌面端/移动端安装引导**：用户如果在同一个浏览器里使用工具 C 超过 2 次（借助 localStorage 埋点计次），界面顶部会优雅滑出：“添加到桌面，下次打开更方便 [立即安装]”。由于 PWA 没有浏览器地址栏边缘，体验等同于原生 App，且零分发成本。这对后续将免费用户转化为 Pro 付费订阅的留存率至关重要。
